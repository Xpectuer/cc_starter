# Requirements: Codex Support

## 1. Problem Statement

cct currently only supports launching Claude Code CLI. Users who also use OpenAI Codex CLI need a separate workflow to manage and launch codex sessions. Issue #6 requests first-class codex support so cct can serve as a unified launcher for both tools.

## 2. Desired Outcome

cct becomes a dual-backend TUI launcher. Users can create profiles for both `claude` and `codex` in a single config file, with backend-specific field validation, a tabbed/grouped UI to visually separate the two, and correct arg-building + exec-replace for each backend.

## 3. Constraints
- **Tech stack**: Rust, ratatui, serde/toml, toml_edit — no new major dependencies
- **Hard boundaries**: Single config file (`profiles.toml`); each profile must specify its backend; backward compatibility for existing claude-only configs (default backend = "claude" if omitted)

## 4. Scope
### In Scope
- New `backend` field on Profile struct (`"claude"` | `"codex"`, default `"claude"`)
- Backend-specific field validation at config parse time (e.g., codex profiles reject `skip_permissions`; claude profiles reject `full_auto`)
- TUI tab/group switching to separate claude and codex profiles
- Codex-specific detail panel showing codex-relevant fields
- Add-form adaptation for codex profiles (different field set)
- `full_auto` boolean field for codex profiles → maps to `--full-auto` flag
- `launch::exec_codex()` — build codex CLI args and exec-replace
- Codex env vars in `[profiles.env]` (e.g., `OPENAI_API_KEY`)
- Unit tests for config parsing (codex profile deserialization, field validation)
- Unit tests for codex arg building
- Integration/E2E tests for the full flow (create codex profile → display in TUI → launch)

### Out of Scope
- Codex auto-install when binary is missing (deferred)
- `sandbox`, `ask_for_approval`, and other codex flags beyond `full_auto`
- Generic CLI tool abstraction (no trait object for backends)
- Model field env-var auto-generation for codex (codex uses its own config system)

## 5. Acceptance Criteria
- [ ] `profiles.toml` with a codex profile (backend = "codex") is parsed correctly
- [ ] Invalid field combinations are rejected at parse time (e.g., codex + skip_permissions)
- [ ] TUI shows tab/group separation between claude and codex profiles
- [ ] Selecting a codex profile and pressing Enter exec-replaces with `codex <args>`
- [ ] `full_auto = true` in a codex profile produces `codex --full-auto` on launch
- [ ] Codex env vars from `[profiles.env]` are injected into the exec environment
- [ ] Unit tests pass for codex config parsing and arg building
- [ ] Integration tests pass for the create-display-launch flow
- [ ] Existing claude-only profiles continue to work without modification
