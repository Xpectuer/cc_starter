# Requirements: Config Add Functionality for cct

## 1. Problem Statement

Currently, adding a new profile to `cct` requires manually editing `~/.config/cc-tui/profiles.toml` in a text editor. Users must know the exact TOML syntax, field names, and the correct set of environment variables for third-party API providers. This creates friction for new users and slows down experienced users who want to quickly set up a new Claude Code profile, especially for non-Anthropic backends.

## 2. Desired Outcome

Users can create new profiles through two intuitive paths:
1. **CLI subcommand** (`cct add`): An interactive prompt-by-prompt flow in the terminal that collects profile details and appends to the config file.
2. **In-TUI inline form** (press `a`): A form within the ratatui TUI where users type profile fields into text inputs, confirm, and save — without leaving the TUI.

Both paths collect five fields:

| Field | Required | Maps to |
|-------|----------|---------|
| Name | yes | `name` |
| Description | no | `description` |
| Base URL | no | `[profiles.env] ANTHROPIC_BASE_URL` |
| API Key | no | `[profiles.env] ANTHROPIC_API_KEY` |
| Model | no | `[profiles.env]` model env vars (see below) |

When **Model** is provided, the following env vars are auto-populated with the model name as the value (user-replaceable):

| Env Var | Default Value |
|---------|---------------|
| `ANTHROPIC_MODEL` | `<model>` |
| `ANTHROPIC_SMALL_FAST_MODEL` | `<model>` |
| `ANTHROPIC_DEFAULT_SONNET_MODEL` | `<model>` |
| `ANTHROPIC_DEFAULT_OPUS_MODEL` | `<model>` |
| `ANTHROPIC_DEFAULT_HAIKU_MODEL` | `<model>` |
| `API_TIMEOUT_MS` | `600000` |
| `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` | `1` |

Advanced fields (skip_permissions, extra_args, additional env) remain editor-editable via `e`.

## 3. Constraints
- **Tech stack**: Rust, ratatui, crossterm, serde/toml, clap. Add `serde::Serialize` derive to `Profile` struct for TOML writing.
- **Hard boundaries**:
  - Must not break existing TUI launch flow
  - Must not corrupt existing `profiles.toml` content
  - Config file format remains backward-compatible (same `[[profiles]]` + `[profiles.env]` TOML structure)

## 4. Scope
### In Scope
- `cct add` CLI subcommand with interactive prompts (name → description → base_url → api_key → model → confirm → save)
- In-TUI inline form triggered by pressing `a` keybind with 5 fields
- Duplicate profile name validation (reject if name exists)
- Confirmation step before writing to disk (both flows)
- Auto-select newly added profile in the TUI list
- Auto-populate `[profiles.env]` section with base_url, api_key, and model-derived env vars
- Append new `[[profiles]]` block (with optional `[profiles.env]`) to existing `profiles.toml`

### Out of Scope
- Editing existing profiles (edit remains via `$EDITOR` on `e` key)
- Deleting profiles
- Model string validation against a known model list
- Advanced fields in the add flow (skip_permissions, extra_args, custom env)
- Profile reordering
- Import/export profiles

## 5. Acceptance Criteria
- [ ] `cct add` prompts for name, description, base_url, api_key, model interactively
- [ ] `cct add` rejects duplicate profile names with a clear error message
- [ ] `cct add` shows a summary and asks for confirmation before saving
- [ ] `cct add` appends a valid `[[profiles]]` block with `[profiles.env]` to `profiles.toml`
- [ ] Pressing `a` in the TUI opens an inline form for name, description, base_url, api_key, model
- [ ] In-TUI form validates for duplicate names before saving
- [ ] In-TUI form shows confirmation before writing to disk
- [ ] After saving via TUI form, cursor auto-selects the new profile
- [ ] Existing profiles are not modified or corrupted when a new one is appended
- [ ] When model is provided, all 5 model env vars + `API_TIMEOUT_MS` + `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` are auto-populated
- [ ] When base_url is provided, `ANTHROPIC_BASE_URL` is set in `[profiles.env]`
- [ ] When api_key is provided, `ANTHROPIC_API_KEY` is set in `[profiles.env]`
- [ ] `clap` handles `cct` (no subcommand → TUI) and `cct add` (→ interactive add flow)
- [ ] `cct --help` shows available subcommands
