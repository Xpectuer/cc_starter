# KISS — Keep It Simple, Stupid

## 原则

优先选择**最简单的可行方案**。复杂度是 bug 的温床，也是维护负担的根源。每一个新增的抽象、新增的分支、新增的配置项，都必须有充分的理由。

## 设计决策

- **不提前设计**：只为当前需求建模，不为假想的未来需求留口子。
- **不过度抽象**：三处重复代码才考虑提取，单次使用的逻辑直接内联。
- **不增加配置项**：能写死的先写死，能用约定替代的不用配置。
- **exec 不 spawn**：`cct` 用 `exec` 替换进程而非 `spawn` 子进程，因为这是最简单的正确行为，无需管理生命周期。
- **五模块扁平结构**：`config / app / ui / launch / cli` 无共享可变状态，职责单一，不引入额外分层。

## 代码风格

- **短函数**：一个函数只做一件事，超过 40 行考虑拆分。
- **无防御性代码**：只在系统边界（用户输入、外部文件、CLI 参数）做校验，内部调用信任类型系统。
- **无向后兼容补丁**：废弃的代码直接删除，不留 `// removed` 注释或空壳。
- **无多余注释**：逻辑自明时不写注释；只有在"为什么这样做"不显而易见时才写。
- **错误处理扁平化**：用 `?` 传播错误，不在每一层都 wrap 新类型，除非类型边界有意义。

## 判断标准

> 当你在两种方案之间犹豫时，选更笨的那个。

如果一个改动让新人读代码需要额外 30 秒，就值得重新考虑。

## 反例（本项目中曾出现）

| 反例 | 正确做法 |
|------|---------|
| 为"将来可能支持多后端"引入 trait object | 直接调用函数，等真正有第二个后端时再抽象 |
| 每个错误分支都单独定义错误类型 | 用 `anyhow::Error` 或 `String` 贯穿，只在公共 API 边界定义具名错误 |
| 用 `Arc<Mutex<>>` 共享 App 状态 | `App` 结构体在主循环中单线程持有，无需同步原语 |
